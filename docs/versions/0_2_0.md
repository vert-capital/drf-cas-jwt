# DRF CAS JWT 0.2.0

## 1. Login CAS gera access e refresh token
**Funcionamento atual:** após login bem-sucedido no CAS, o sistema cria um `RefreshToken` e deriva um access token JWT.

**Por que está implementado:** permitir autenticação stateless nas APIs DRF sem depender da sessão web do Django.

**Exemplo:** usuário faz login via CAS e recebe tokens para consumir endpoints protegidos da API.

## 2. Identificação de dispositivo via User-Agent
**Funcionamento atual:** no login e na autenticação, o sistema coleta informações de device/browser/OS com `django-user-agents`.

**Por que está implementado:** criar uma associação básica entre token e contexto do cliente para reduzir reuso simples em ambiente diferente.

**Exemplo:** token emitido em Chrome Windows é validado apenas quando o request apresenta a mesma família de navegador e sistema operacional cadastrados.

## 3. Cadastro de Device por usuário
**Funcionamento atual:** se não existir registro equivalente para o usuário, um novo `Device` é criado; caso exista, ele é reutilizado.

**Por que está implementado:** manter histórico mínimo de contexto de acesso por usuário sem duplicar registros para o mesmo perfil de cliente.

**Exemplo:** primeiro acesso pelo celular cria um `Device`; acessos seguintes pelo mesmo perfil reaproveitam esse cadastro.

## 4. Persistência do token com hash e IP
**Funcionamento atual:** o access token é convertido em hash MD5 e salvo no modelo `Token` junto com o IP.

**Por que está implementado:** evitar armazenar token em texto puro no banco e permitir validação adicional no backend.

**Exemplo:** em vez de salvar o JWT completo, salva-se somente seu hash para comparação durante autenticação.

## 5. Um token ativo por device
**Funcionamento atual:** no login, tokens anteriores do mesmo `Device` são removidos (soft delete) antes de salvar o novo token.

**Por que está implementado:** reduzir sessões simultâneas por dispositivo e simplificar estratégia de invalidação.

**Exemplo:** ao relogar no mesmo notebook, o token antigo deixa de ser aceito e somente o novo permanece válido.

## 6. Autenticação JWT com validação de vínculo
**Funcionamento atual:** além da validação padrão do JWT, o backend exige existência de `Token` com hash correspondente e device compatível.

**Por que está implementado:** adicionar camada de segurança para bloquear uso de token fora do contexto esperado.

**Exemplo:** um JWT válido criptograficamente, mas sem registro vinculado ao device, é rejeitado com `AuthenticationFailed`.

## 7. Logout remove token persistido
**Funcionamento atual:** no logout, o token do header `Authorization` é hasheado e removido da tabela `Token`.

**Por que está implementado:** invalidar no backend o token em uso, mesmo que ainda não tenha expirado por tempo.

**Exemplo:** usuário efetua logout e tentativas de reutilizar o mesmo access token passam a falhar.

## 8. Redirecionamento para frontend com tokens
**Funcionamento atual:** fora do admin, após login o sistema redireciona para `FRONTEND_AUTH_REDIRECT` incluindo access e refresh no caminho da URL.

**Por que está implementado:** simplificar integração entre CAS (backend) e aplicação frontend no momento de bootstrap da sessão.

**Exemplo:** frontend recebe os tokens na rota de callback e inicializa estado de autenticação do usuário.
