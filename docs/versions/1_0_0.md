# DRF CAS JWT 1.0.0

## 1. Remover tokens da URL de redirecionamento
**Sugestão:** não transportar access/refresh token em path ou query string.

**Por que implementar:** URLs vazam em logs, histórico do browser, analytics e referer header. Expõe tokens permanentemente.

**Implementação:** usar cookie `HttpOnly` + `Secure` + `SameSite` para refresh token; access token em header ou sessionStorage.

**Exemplo antes:**
```
GET http://frontend.com/callback/eyJ0eXAiOiJKV1QiLCJhbGc.../refresh_token
```

**Exemplo depois:**
```
GET http://frontend.com/callback
Set-Cookie: refresh_token=....; HttpOnly; Secure; SameSite=Strict
Body: { "access_token": "...", "user": {...} }
```

## 2. Substituir MD5 por HMAC-SHA256 no armazenamento
**Sugestão:** trocar hash MD5 do token para HMAC-SHA256 com segredo do servidor.

**Por que implementar:** MD5 é criptograficamente quebrado; HMAC oferece integridade + autenticação do hash.

**Implementação:** salvar `HMAC-SHA256(server_secret, token)` no banco; comparar com `compare_digest()` para evitar timing attacks.

**Exemplo antes:**
```python
token_hash = hashlib.md5(str(jwt_token).encode()).hexdigest()
# Vulnerável: MD5 reversível em tabelas rainbow
```

**Exemplo depois:**
```python
server_secret = settings.SECRET_KEY
token_hash = hmac.new(
    server_secret.encode(),
    str(jwt_token).encode(),
    hashlib.sha256
).hexdigest()
# Seguro: impossível recriar sem SECRET_KEY
```

## 3. Vincular refresh token ao banco (jti tracking)
**Sugestão:** persistir `jti` (JWT ID) do refresh token para rastrear e revogar cadeia de tokens.

**Por que implementar:** access token expira em 5-15 min, mas refresh pode ser válido por semanas. Refresh comprometido = acesso indefinido.

**Implementação:** tabela `RefreshTokenFamily` rastreia `jti` do refresh, usuário, device (UA), IP, data de criação.

**Exemplo:**
```python
class RefreshTokenFamily(models.Model):
    jti = models.CharField(max_length=255, unique=True, db_index=True)
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    device_family = models.CharField(max_length=64)  # user-agent family
    ip_address = models.GenericIPAddressField()
    created_at = models.DateTimeField(auto_now_add=True)
    revoked_at = models.DateTimeField(null=True, blank=True)
    
    class Meta:
        indexes = [
            models.Index(fields=['user', 'revoked_at']),
        ]
```

## 4. Implementar refresh rotation com detecção de reuse
**Sugestão:** a cada troca de refresh token, invalidar anterior; se antigo reaparece = sessão comprometida.

**Por que implementar:** reduz janela de impacto de vazamento; detecta tentativa de replay.

**Implementação:** ao renovar refresh, criar novo `jti` e marcar anterior como "rotated"; se `jti` revoked reaparece = revogar toda cadeia do usuário.

**Exemplo:**
```
Dia 1: refresh_A gerado (jti=abc)
Dia 2: refresh_A → refresh_B (jti=def), A marcado como "rotated"
Dia 3: refresh_A reaparece (jti=abc)
       → Detector: ABC foi rotated mas reapareceu
       → Ação: Revogar todos os tokens do usuário + notificar
```

## 6. Validação segura de token em header
**Sugestão:** rejeitar token sem `Authorization` header; usar `Bearer` scheme; fallback seguro em erro.

**Por que implementar:** aceitar token de múltiplas fontes (URL, cookie, header) aumenta superfície de ataque.

**Implementação:** aceitar APENAS `Authorization: Bearer <token>`; rejeitar query param, cookie sem flag.

**Exemplo antes:**
```python
token = request.GET.get('token') or request.headers.get('Authorization')
# Aceita múltiplas fontes, XSS pode injetar URL
```

**Exemplo depois:**
```python
auth_header = request.headers.get('Authorization', '')
if not auth_header.startswith('Bearer '):
    raise AuthenticationFailed("Invalid authorization header")
token = auth_header[7:]  # Remove "Bearer "
```

## 7. Rate limiting e detecção de anomalia
**Sugestão:** limitar tentativas de refresh por usuário/IP; flagear mudanças brustas de contexto.

**Por que implementar:** reduz impacto de força bruta; detecta roubo antes de dano maior.

**Implementação:** cache Redis com chave `refresh_attempts:{user_id}:{ip}`; após N tentativas em M segundos, bloquear.

**Exemplo:**
```
Limite: 5 refreshes/min por usuário
Violação: bloquear por 15 min + alertar usuário via email
Contexto: se mudança de país em <1h + novo IP = extra validação (MFA)
```

## 8. Melhorar rastreabilidade com audit log mínimo
**Sugestão:** registrar eventos críticos: login, logout, refresh, rejeição com razão, revogação.

**Por que implementar:** facilita investigação de incidente; identifica padrões de ataque.

**Implementação:** tabela `TokenAuditLog` com user, evento, device_family, ip, timestamp, motivo_rejeição.

**Exemplo:**
```
2026-06-08 14:23:15 | user_id=123 | event=LOGIN | device=Chrome | ip=200.1.2.3
2026-06-08 14:25:30 | user_id=123 | event=REFRESH_DENIED | reason=REUSE_DETECTED | ip=10.0.0.1
2026-06-08 14:26:00 | user_id=123 | event=LOGOUT | device=Chrome | ip=200.1.2.3
```

## 10. Configuração e compatibilidade
**Sugestão:** adicionar flags no `settings.py` para ativar/desativar features; rollout gradual.

**Por que implementar:** permite migração sem quebra de clientes existentes.

**Configurações:**
```python
DRF_CAS_JWT = {
    "STRICT_REFRESH_ROTATION": True,        # v1.0 default
    "HMAC_ALGORITHM": "sha256",              # trocar de MD5
    "REFRESH_TOKEN_LIFETIME": timedelta(days=7),
    "ACCESS_TOKEN_LIFETIME": timedelta(minutes=15),
    "TOKENS_IN_URL_DEPRECATED": True,        # warning, depois erro
    "LOGOUT_REQUIRES_POST": True,
    "AUDIT_LOG_ENABLED": True,
    "RATE_LIMIT_REFRESH": {"attempts": 5, "window": 60},
}
```
